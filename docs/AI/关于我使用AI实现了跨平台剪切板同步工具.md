# 我用 AI 写了一个跨平台剪切板同步工具

## 前言

最近一直在用多台设备办公 —— Linux 做开发、Windows 偶尔用、Android 手机随身带。一个高频痛点是：**在一台设备上复制的内容，没法直接粘贴到另一台设备上**。

市面上的方案要么依赖特定生态（比如 Apple 的 Universal Clipboard），要么需要把数据传到第三方服务器，要么不支持图片/文件。于是我决定自己搞一个，但有个前提 —— **我不想一行行写代码，我想看看 AI 能不能帮我从零搞定整个项目**。

最终成果：一个支持 **Windows、Linux、Android 三端**的剪切板同步工具，支持**文字、图片、文件**同步，具备 **P2P 直连、NAT 打洞、服务器中转**三级传输策略。核心代码约 **7500 行**，从第一次提示词到基本可用，前后大约 **2 天**，实际上是两个晚上，每天晚上一两个小时。

项目地址: [https://github.com/JoeKerouac/SyncClipboard](https://github.com/JoeKerouac/SyncClipboard)

## 架构概览

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  Windows │     │  Linux   │     │ Android  │
│  Client  │     │  Client  │     │  Client  │
│   (C)    │     │   (C)    │     │ (Kotlin) │
└────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │
     │    WebSocket   │    WebSocket   │
     └────────┬───────┘────────┬───────┘
              │                │
         ┌────┴────────────────┴────┐
         │     Server (Java 17)     │
         │   Spring Boot WebSocket  │
         └──────────────────────────┘
```

- **服务端**：Java 17 / Spring Boot，负责认证、消息转发、P2P 协调
- **Linux / Windows 客户端**：纯 C，依赖 libwebsockets + OpenSSL
- **Android 客户端**：Kotlin，前台服务 + 无障碍服务保活

文字内容通过 WebSocket 直接转发（AES-256 端到端加密，服务端不解密）；图片和文件则通过 P2P 传输：

1. **局域网直连** —— 同网段时直接 TCP 传输
2. **NAT 打洞** —— 不在同网段时由服务端协调 UDP 打洞
3. **服务端中转** —— 以上都失败且文件较小时，走服务端转发

## 用 AI 生成项目的过程

整个项目并不是一次提示词就搞定的，而是经过了 **4 轮迭代**，每一轮都是在上一轮的基础上提出新的需求或修复问题。下面是我使用的提示词的演进过程（完整内容见仓库中的 `main.prompt` 文件）。

### 第一轮：搭骨架

第一轮提示词定义了整个项目的基本框架：

> 帮我在当前目录实现剪切板多端共享，需要分为以下模块：
> - Windows模块、Android模块、Linux模块、server模块
>
> 各端需要实现剪切板监听、同步，跟服务端建立长连接……
>
> 约束：服务端使用 Java 17；Windows、Linux 尽量使用 C

这一轮 AI 直接生成了：
- 完整的 Java Spring Boot 服务端（WebSocket 通信、用户认证、消息转发）
- C 语言的 Linux/Windows 客户端骨架（libwebsockets 长连接、AES 加密）
- Kotlin 的 Android 客户端（基本 UI + WebSocket 通信）
- 统一的 JSON 通信协议

**产出**：基本的文字剪切板同步已经可以工作了。

### 第二轮：优化 Android

第一轮生成的 Android 端在后台时无法监听剪切板变更，于是第二轮补充：

> 利用无障碍服务，在后台监听剪切板变更……
> 系统休眠状态下，使用 WorkManager 定时同步……

AI 加入了无障碍服务、前台服务保活、WorkManager 定时同步等机制。

### 第三轮：图片和文件传输（核心功能）

这是工作量最大的一轮。纯文字同步简单，但图片和文件不能直接走服务端中转（太慢、带宽有限），需要 P2P 传输：

> 图片不应该直接经过服务器传输，而是通过 P2P 传输：
> 1. 优先局域网直接传输
> 2. 不在同一局域网时，通过 NAT 打洞建立连接
> 3. 以上都不行且文件小于 300K，走服务器中转
> 4. 否则放弃
>
> 注意：两个客户端可能一个在另一个的子网……NAT 打洞是有方向的……

这一轮 AI 设计并实现了完整的 P2P 文件传输协议：
- `file_offer` / `file_notify` / `file_request` / `file_peer_info` 多步握手
- LAN 直连 + NAT 打洞 + Relay 中转的三级 fallback
- HMAC-SHA256 认证防止错误连接
- 跨平台的 socket 抽象层（`common/file_transfer.c`）

### 第四轮：打磨细节

解决实际使用中发现的问题：

> 1. Linux 端复制时屏幕会闪一下
> 2. 传输的文件应该存到系统临时目录
> 3. Android 连接不唯一
> 4. 文件传输开关（服务端 + 客户端分级控制）

## 提示词的经验总结

回顾整个过程，有几点经验：

### 1. 分层提需求，别一口吃成胖子

第一轮先把骨架搭好（通信协议、基本同步），确认能跑了再加功能（图片、文件、P2P）。如果把所有需求塞到一个提示词里，AI 容易顾此失彼。

### 2. 约束条件很重要

明确告诉 AI "服务端用 Java"、"客户端用 C"，这样生成的代码技术栈一致，不会出现一半 Python 一半 Go 的混乱局面。

### 3. 描述场景比描述实现更好

比如 NAT 打洞的需求，我没有说"用 STUN 协议"，而是描述了场景：

> 两个客户端可能一个在另一个的子网，从 A 到 B 不通但 B 到 A 可以通

AI 根据场景自己选择了合适的实现方案。

### 4. AI 生成的代码需要大量调试

虽然 AI 生成了大部分代码，但实际跑起来会遇到各种问题。比如：
- X11 剪切板读取在 64 位系统上数据截断
- P2P 认证协议 Android 端和 C 端不一致
- 多线程竞争导致的 `listen_fd` 被覆写（全局状态 `g_xfer` 被并发修改）
- 多客户端场景下只有第一个接收方能收到文件

这些问题有的是 AI 自己修复的（通过提供日志让 AI 分析），有的需要人工介入引导。**AI 写代码很快，但调 bug 还是需要人的判断力。**

## 哪些是 AI 做得好的

- **协议设计**：WebSocket JSON 消息协议、P2P 握手流程，AI 一次性设计得相当完整
- **跨平台抽象**：`file_transfer.c` 中用宏和条件编译兼容 Linux/Windows 的 socket API，做得很干净
- **构建系统**：CMakeLists.txt、Gradle 配置、MinGW 交叉编译脚本，都是一次生成直接能用
- **代码量**：7500 行跨 4 种语言的代码，如果纯手写可能需要数周，AI 几轮对话就出来了

## 哪些是 AI 做不好的

- **并发和状态管理**：全局 `g_xfer` 的竞态问题 AI 没有在第一时间意识到，调了好几轮才定位到
- **平台特性差异**：X11 的 `Atom` 在 64 位系统上是 8 字节这种细节，AI 最初没处理对
- **端到端集成测试**：AI 可以编译通过，但多设备联调的问题需要真实环境反馈

## 结语

这个项目让我确信：**AI 已经可以作为"初级开发者"来使用了** —— 给它一个清晰的需求描述，它能快速产出可工作的代码骨架。但它还不是"高级开发者"，并发问题、边界情况、跨平台细节，这些仍然需要人类工程师的经验和判断。

最好的工作模式是：**人负责架构设计和需求拆解，AI 负责编码实现，人再做 code review 和集成调试**。这种协作方式下，开发效率确实可以提升数倍。

如果你也有类似的多端同步需求，欢迎试用或参考这个项目。所有代码（包括生成它的提示词）都在 GitHub 上开源，原提示词也在项目的git仓库中，参考 `main.prompt` 文件。
